## 1. 演示目标与时长

### 1.1 要向评委证明什么

| 作业要求 | 网页上如何体现 |
|----------|----------------|
| **C6** 云边集成 | Edge `POST /predict` → Agent `POST /recommend` 全链路 |
| **C7** 交互原型 | 一键故障注入 + 流水线四格 + 事件时间线 / 详情 |
| **可解释性** | Recommendation 文本、RAG 引用、Reasoning trace 表格 |
| **优雅降级** | 「Edge-only Degrade」预设：Agent 跳过，Edge 仍告警 |

### 1.2 建议总时长

| 段落 | 时长 |
|------|------|
| 准备（后台起服务） | 5～8 分钟（可提前完成） |
| 浏览器内正式演示 | **10～15 分钟** |
| 评委提问 + 手动注入 | 预留 3～5 分钟 |

### 1.3 30 秒电梯陈述（开场可用）

**English**

> AgentPV is a cloud–edge intelligent O&M system for PV plants and BESS. A quantised 1D-CNN on the edge classifies 60-second sensor windows in milliseconds; a ReAct agent with RAG and tools turns structured alerts into auditable maintenance recommendations. This dashboard is our Component-7 operator prototype: one click injects a fault and shows the full pipeline end to end.

**中文**

> AgentPV 是面向光伏与储能的云边协同智能运维系统。边缘用量化 1D-CNN 在毫秒级完成 60 秒窗口的多类故障分类；云端 ReAct 智能体结合 RAG 与工具，把结构化告警变成可审计的维护建议。本页是作业 Component 7 的操作员原型：一键注入故障，展示从仿真到落盘的全链路。

## 4. 正式演示流程（逐步 + 双语旁白）

> **节奏建议**：全链路注入后 **等 spinner 结束** 再讲下一步；两次全链路之间 **间隔 10 秒以上**，避免 Agent 排队。单次 Agent 调用通常 **8～20 秒**（Ollama + ReAct），HTTP 超时默认 **120 秒**。

---

### 步骤 1 · 确认服务在线（约 1 分钟）

**操作**

1. 看 **Edge service**、**Agent service** 指标是否为 **Online**。
2. 若为 **Offline**，对照 [§8 常见问题](#8-常见问题速查) 处理，**不要**开始全链路注入。

**English narration**

> Before any injection, both services must show Online. Edge handles sub-100-millisecond inference; Agent handles RAG retrieval and the ReAct loop. If Agent is offline, we can still demo edge-only degradation later.

**中文旁白**

> 注入前必须确认 Edge 与 Agent 均为 Online。Edge 负责亚 100 毫秒级推理；Agent 负责 RAG 与 ReAct。若 Agent 离线，稍后仍可用「仅边缘降级」演示。

---

### 步骤 2 · 介绍一键场景与流水线（约 1 分钟）

**操作**

1. 鼠标依次经过五个按钮（**不要点击**）：
   - **PV Inverter Fault** — critical，全链路
   - **PV Partial Shading** — warning，全链路
   - **BESS Thermal Anomaly** — critical，全链路
   - **PV Normal** — monitor，**不触发 Agent**（severity 规则）
   - **Edge-only Degrade** — critical，**跳过 Agent**
2. 指向下方四步表格 **Simulation → Edge → Agent → Persist**。

**English narration**

> These five presets match our scripted Component-7 benchmark. Full pipeline runs simulation, edge classification, and—when severity is warning or critical—the cloud agent. Normal-class windows stop at the edge by design, matching our orchestrator trigger rule. Edge-only degrade skips the agent entirely to show graceful degradation.

**中文旁白**

> 五个预设与 C7 脚本一致。全链路走仿真、边缘分类；严重度为 warning 或 critical 时才调用智能体。Normal 类按设计不触发 Agent，与编排器规则一致。「Edge-only Degrade」故意跳过 Agent，用于展示降级。

---

### 步骤 3 · 主场景：PV 逆变器严重故障（约 2～4 分钟）

**操作**

1. 点击 **「PV Inverter Fault」**。
2. 等待页面 spinner：**Injecting PV Inverter Fault… / 正在注入…** 结束（勿连点其他按钮）。
3. 查看 **Latest injection result · 最近一次注入结果** 区域。

**预期结果（四格）**

| 格 | 预期 | 含义 |
|----|------|------|
| 1 Simulation window | ✅ | 60 s 窗口 + 真值标签 `Inverter_fault` |
| 2 Edge /predict | ✅ | 延迟通常 **&lt;30 ms**，severity **critical** |
| 3 Agent /recommend | ✅ | 延迟约 **8～20 s**，有 urgency |
| 4 Persist JSONL | ✅ | 已写入侧栏指定 JSONL |

**指着屏幕讲解（按顺序）**

1. 标题行：`system_id`、Ground truth、Predicted、红色 **CRITICAL** 徽章  
2. 指标：**Edge latency**、**Agent latency**、**Confidence**、**KB citations**  
3. 左栏 **Agent recommendation**：自然语言 `recommended_action`  
4. 左栏 **RAG sources**：`kb_*.md` 等引用  
5. 右栏 **Edge alert**：JSON 与 `docs/alert_schema.json` 契约一致  

**English narration**

> We inject a labelled inverter-fault window. Step one builds a 60-second tensor from the simulator. Step two sends it to the edge ONNX model, which returns a critical alert in tens of milliseconds. Step three forwards that alert to the ReAct agent, which retrieves playbook chunks and returns a structured recommendation with citations—this is the interpretability requirement from Component 4. Step four appends an orchestrator-equivalent event to JSONL for audit.

**中文旁白**

> 我们注入带标签的逆变器故障窗口。第一步仿真生成 60 秒张量；第二步边缘 ONNX 在毫秒级返回 critical 告警；第三步将告警送入 ReAct 智能体，检索运维文档并返回带引用的维护建议，满足 Component 4 的可解释性要求；第四步写入 JSONL，供审计与时间线展示。

**中文旁白**

> 事件详情页对应作业的可解释性要求。Alert JSON 是边缘与云端的固定契约；Recommendation 展示紧迫性、置信度与 grounded 引用；Reasoning trace 表格是 ReAct 审计轨迹，每一行对应观察、推理、行动或反思，便于运营或监管复盘。

## 6. 失败时的降级讲法

若 **步骤 3** 第三步为 **❌**（Agent 失败）而 Edge 为 **✅**：

**English**

> The edge alert is still valid—that is the graceful-degradation story. The agent step failed, likely due to Ollama load or timeout; we document this in our integration report. Let me show Edge-only degrade and the edge latency budget, which meets the assignment’s sub-100-millisecond requirement.

**中文**

> 边缘告警仍然有效，这正是优雅降级要说明的。Agent 失败多因 Ollama 负载或超时，集成报告中有记录。接下来演示 Edge-only 降级与边缘延迟预算，满足作业亚 100 毫秒要求。

## 8. 常见问题速查

| 现象 | 可能原因 | 处理 |
|------|----------|------|
| Agent 第三步 ❌，延迟约 **120000 ms** | HTTP **120 s** 超时 | 确认 Ollama、`8001` 已就绪；勿与编排器/压测并行；先演示 Edge-only |
| Edge / Agent 显示 **Offline** | 8000 / 8001 未起 | 启动 uvicorn；端口占用则复用已有进程 |
| 注入成功但 Tab 无新行 | 未 **Refresh** 或 path 不一致 | 点 Refresh；核对 Event log path |
| 一键按钮后无结果区 | 参数校验失败 | 看顶部红色 error；检查 fault 与 system 是否匹配 |
| Streamlit 打不开 | 8501 被占 | 换端口：`streamlit run dashboard/app.py --server.port 8502` |
| 与 Docker 冲突 | 同时 compose up | 答辩只用本机三终端；Compose 前 `docker compose down` |
